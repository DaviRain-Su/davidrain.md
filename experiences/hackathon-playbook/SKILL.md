# Hackathon 实战手册

从赛前准备到 Demo Day 的完整流程和技巧。

---

## 一、时间线规划

```
T-7天 (赛前一周)
    ├── 团队组建
    ├── 技术栈确认
    └── 环境准备

T-3天 (赛前三天)
    ├── 需求锁定
    ├── 架构设计
    └── 任务拆分

T-1天 (赛前一天)
    ├── 开发环境验证
    ├── 账号/密钥准备
    └── 休息准备

T-0 (比赛开始)
    ├── 0-4h: 核心合约开发
    ├── 4-8h: 前端基础
    ├── 8-16h: 集成测试
    ├── 16-20h: 完善功能
    └── 20-24h: Demo 准备
```

---

## 二、赛前准备清单

### 团队组建

- [ ] **技术负责人** (1人): 架构决策、代码审查、技术难题
- [ ] **合约开发者** (1-2人): 智能合约开发、测试
- [ ] **前端开发者** (1人): UI/UX、合约交互
- [ ] **产品经理** (1人，可兼职): 需求定义、Demo 准备、Pitch

**理想团队规模**: 3-4 人

### 技术栈确认

```yaml
# Web3 Stack Template
blockchain:
  - Sui / Ethereum / Solana  # 根据比赛要求
  
contract_language:
  - Move / Solidity / Rust
  
frontend:
  - Next.js 14 + TypeScript
  - Tailwind CSS
  - @mysten/dapp-kit  # Sui
  
storage:
  - Walrus / IPFS / Arweave  # 去中心化存储
  
Tools:
  - Git + GitHub
  - Vercel (部署)
  - Figma (设计)
```

### 环境准备

- [ ] 所有成员安装开发环境
- [ ] 创建共享 GitHub Repo
- [ ] 设置 Vercel 自动部署
- [ ] 准备测试网代币 (Faucet)
- [ ] 创建共享文档 (Notion/Google Doc)

## 三、比赛日流程

### Phase 1: 核心合约 (0-4小时)

**目标**: 跑通最关键的智能合约

**执行步骤**:
1. **快速设计** (30min)
   - 白板画出核心流程
   - 确定最少必要功能
   - 使用第一性原理删除非核心功能

2. **合约开发** (2.5h)
   ```move
   // 优先级：核心功能 > 边界情况 > 优化
   // 示例：任务市场的最小版本
   
   // 1. 创建任务
   public entry fun create_task(reward: Coin<SUI>, description: String)
   
   // 2. 接受任务
   public entry fun accept_task(task: &mut Task)
   
   // 3. 完成任务
   public entry fun complete_task(task: &mut Task, proof: String)
   
   // 4. 支付奖励
   public entry fun claim_reward(task: &mut Task, ctx: &mut TxContext)
   ```

3. **部署测试** (1h)
   - 部署到测试网
   - 至少完成一次端到端测试
   - 记录合约地址

**检查点**:
- [ ] 合约编译通过
- [ ] 核心功能可调用
- [ ] 至少有一个成功的交易记录

### Phase 2: 前端基础 (4-8小时)

**目标**: 用户可以连接钱包并看到数据

**执行步骤**:
1. **项目初始化** (30min)
   ```bash
   npx create-next-app@latest my-app --typescript --tailwind --eslint
   cd my-app
   npm install @mysten/dapp-kit @mysten/sui
   ```

2. **基础组件** (2h)
   - 连接钱包按钮
   - 显示账户信息
   - 基础布局

3. **合约交互** (2h)
   - 读取合约数据
   - 发送交易
   - 显示交易状态

4. **简单 UI** (1.5h)
   - 任务列表
   - 创建任务表单
   - 基础样式

**检查点**:
- [ ] 可以连接钱包
- [ ] 可以读取链上数据
- [ ] 可以发送交易

### Phase 3: 集成测试 (8-16小时)

**目标**: 核心流程完整跑通

**执行步骤**:
1. **端到端测试** (2h)
   - 完整用户流程
   - 边界情况处理
   - 错误提示

2. **Bug 修复** (4h)
   - 记录所有 Bug
   - 按优先级修复
   - 保持核心功能稳定

3. **UI 完善** (2h)
   - 加载状态
   - 错误提示
   - 移动端适配

**检查点**:
- [ ] 完整流程可以跑通
- [ ] 没有阻碍性 Bug
- [ ] UI 可用

### Phase 4: 功能完善 (16-20小时)

**目标**: 添加亮点功能

**优先级**:
1. **核心功能增强**: 让主要功能更完整
2. **差异化功能**: 与竞品不同的地方
3. **体验优化**: 动画、交互细节

**不要做的事**:
- ❌ 重构代码
- ❌ 添加非核心功能
- ❌ 过度优化

### Phase 5: Demo 准备 (20-24小时)

**目标**: 准备好展示材料

**必须完成**:
- [ ] 部署到生产环境
- [ ] 准备演示脚本
- [ ] 制作 Pitch Deck
- [ ] 录制 Demo 视频（可选但推荐）

## 四、Demo 技巧

### 演示脚本模板

```markdown
# Demo 脚本 (2-3 分钟)

## Hook (15s)
"大家好，我们是 [Team Name]。今天展示的是 [项目名]，
一个让 AI Agent 可以自主协作和获得收益的平台。"

## Problem (20s)
"目前 AI Agent 虽然可以执行复杂任务，但缺乏：
1. 可信的协作机制
2. 能力验证方式  
3. 收益分配渠道"

## Solution (30s)
"我们的解决方案是 Agent Arena：
- 任务发布者发布任务并质押奖励
- Agent 自主接单并完成任务
- 链上验证和自动结算
- 构建可验证的 Agent 声誉系统"

## Demo (60s)
[现场演示]
1. 发布一个任务（展示创建流程）
2. Agent 接受任务（展示接单）
3. 完成任务并验证（展示验证机制）
4. 领取奖励（展示支付流程）

## Closing (15s)
"Agent Arena 让 AI Agent 经济真正成为可能。
我们是 [Team Name]，谢谢！"
```

### Pitch Deck 结构

1. **封面**: 项目名 + 一句话描述
2. **问题**: 痛点清晰
3. **解决方案**: 简洁有力
4. **产品 Demo**: 截图/流程图
5. **技术架构**: 技术栈 + 创新点
6. **市场**: 目标用户 + 市场规模
7. **团队**: 成员 + 背景
8. **里程碑**: 下一步计划

### Demo 视频要点

- **时长**: 2-3 分钟
- **内容**: 核心功能展示
- **旁白**: 清晰、有节奏
- **字幕**: 增加可访问性

## 五、评审标准应对

### 常见评审维度

| 维度 | 权重 | 应对策略 |
|------|------|----------|
| **技术创新** | 30% | 强调链上创新点，不要只讲 UI |
| **完成度** | 25% | 确保核心功能可用，不要画饼 |
| **实用性** | 20% | 说明真实使用场景 |
| **演示效果** | 15% | 准备流畅的 Demo 脚本 |
| **团队** | 10% | 展示分工和协作 |

### 常见评委问题

1. **"你们的创新点是什么？"**
   - 回答框架：现有方案 + 你们的新方法 + 为什么更好

2. **"怎么防止作弊/攻击？"**
   - 提前准备安全设计说明

3. **"下一步计划是什么？"**
   - 有具体的里程碑和时间线

4. **"为什么选择这个链/技术？"**
   - 技术匹配度 + 生态优势

- [ ] 彩排一次完整 Demo

### 提交前

- [ ] README 包含：项目描述、技术栈、运行说明
- [ ] 提交合约地址
- [ ] 提交 Demo 链接

## 七、常见陷阱

| 陷阱 | 原因 | 解决方案 |
|------|------|----------|
| 功能过多 | 想做太多 | 用第一性原理删减 |
| 完美主义 | 追求代码质量 | Hackathon 先完成再完美 |
| 分工不清 | 没有明确 Owner | 每个功能指定唯一负责人 |
| Demo 失败 | 没有充分测试 | 提前彩排，准备 Plan B |
| 睡眠不足 | 通宵达旦 | 至少保证 4 小时睡眠 |
